# Splunk Technology Add-on (TA) Development Agent

## Goal
Generate production-ready Splunk Technology Add-ons (TAs) using the UCC (Universal Configuration Console) framework from API documentation or specifications. The TA should be ready to pass AppInspect validation and be published to Splunkbase.

## Overview
This agent analyzes API documentation to determine the appropriate Splunk TA components (scripted inputs, custom commands, alert actions, etc.) and generates a complete, working add-on using UCC's configuration-as-code approach.

## What the Developer Provides

**Required Input:**
- API documentation (OpenAPI/Swagger spec, REST API docs, developer guide, or similar)
- Add-on metadata:
  - Add-on name (e.g., `TA-myservice`)
  - Display name (e.g., `My Service Add-on for Splunk`)
  - Brief description of the service/API
  - Author information

**Optional Input:**
- Authentication details (API key, OAuth, basic auth patterns)
- Specific endpoints the developer wants to prioritize
- Data collection frequency preferences
- Splunk CIM (Common Information Model) alignment requirements

## What the Agent Produces

**Final Deliverables:**
1. Complete UCC-based TA package structure with:
   - `globalConfig.json` (the core UCC configuration)
   - Scripted input(s) for data collection
   - Custom SPL command(s) where appropriate
   - Alert action(s) if applicable
   - Configuration UI (auto-generated by UCC)
   - README and documentation
   - Basic test suite
   - AppInspect-compliant structure

2. Build artifacts:
   - Built TA package (via `ucc-gen build`)
   - Installable `.tar.gz` archive (via `ucc-gen package`)

## High-Level Workflow

### Phase 1: API Analysis
1. **Read and understand** the provided API documentation
2. **Identify API characteristics:**
   - Authentication method(s)
   - Available endpoints and their purposes
   - Data formats (JSON, XML, etc.)
   - Rate limiting / pagination patterns
   - Webhook or streaming capabilities

### Phase 2: TA Architecture Design
3. **Decide what components to build** based on API analysis:
   - **Scripted Input:** Create if there's a `/events`, `/logs`, `/alerts`, or similar endpoint that provides time-series data worth indexing
   - **Custom Command:** Create if there are search/query endpoints that analysts would want to call from SPL (e.g., `| myservice_lookup user="john"`)
   - **Alert Action:** Create if the API accepts POST/PUT to trigger actions (e.g., create ticket, send notification)
   - **Configuration Tab:** Always include for account/credential management

4. **Determine authentication pattern:**
   - API key in headers
   - OAuth 2.0 flow
   - Basic authentication
   - Custom token-based auth

### Phase 3: UCC Initialization
5. **Bootstrap the TA structure:**
```bash
   pip install splunk-add-on-ucc-framework
   ucc-gen init --addon-name "TA-myservice" \
                --addon-display-name "My Service Add-on for Splunk" \
                --addon-input-name myservice_input
```

6. **Navigate to the generated structure:**
```
   TA-myservice/
   └── package/
       ├── globalConfig.json    # This is the key file to customize
       ├── bin/
       └── README/
```

### Phase 4: Configuration Development
7. **Edit `globalConfig.json`** to define:
   - **Configuration page** (account credentials, global settings)
   - **Input definitions** (data collection parameters)
   - **Custom commands** (if applicable)
   - **Alert actions** (if applicable)

8. **Write Python helper modules** in `package/bin/`:
   - Modular input script(s) for data collection
   - Custom command handler(s)
   - Alert action handler(s)
   - Shared utilities (API client, error handling, checkpoint management)

### Phase 5: Testing & Validation
9. **Create basic tests:**
   - Unit tests for API client logic
   - Integration tests (if possible with API sandbox/mock)
   - Manual smoke test checklist

10. **Build the TA:**
```bash
    ucc-gen build --source TA-myservice/package --ta-version 1.0.0
```

11. **Validate with AppInspect:**
    - Check for common compliance issues
    - Ensure proper licensing, metadata, and file structure
    - Fix any blocking or critical findings

### Phase 6: Documentation & Packaging
12. **Generate documentation:**
    - README.md with setup instructions
    - Configuration guide
    - Troubleshooting section

13. **Package for distribution:**
```bash
    ucc-gen package --path output/TA-myservice
```

14. **Deliver to developer:**
    - Packaged `.tar.gz` file
    - Installation instructions
    - Configuration examples
    - Known limitations or next steps

## Decision Criteria

### When to Create a Scripted Input
- API has endpoints that return event/log data
- Data should be indexed in Splunk for searching
- Data changes over time (not purely static/reference data)
- Examples: `/events`, `/logs`, `/metrics`, `/alerts`, `/incidents`

### When to Create a Custom Command
- API has search/lookup/enrichment endpoints
- Analysts need to query external data from SPL
- Results should be returned on-demand (not pre-indexed)
- Examples: `/users/{id}`, `/search?query=`, `/threat-intel/lookup`

### When to Create an Alert Action
- API accepts actions that should be triggered by Splunk alerts
- Common patterns: ticket creation, notification, remediation
- Examples: POST to `/tickets`, `/notifications`, `/incidents`

### When to Add OAuth Support
- API documentation explicitly describes OAuth 2.0 flows
- API requires user consent or per-user authorization
- Basic API keys are insufficient

## Key Files to Customize

### `globalConfig.json`
This is the **heart** of the UCC-based TA. It defines:
- **meta:** Add-on name, version, description, author
- **pages/configuration:** Account setup, proxy, logging
- **pages/inputs:** Input configuration UI and parameters
- **alerts:** Alert action definitions (if any)

### `package/bin/<input_name>.py`
The modular input script that:
- Reads configuration from Splunk
- Calls the API
- Formats data as Splunk events
- Handles checkpointing (saves last successful fetch)
- Implements error handling and logging

### `package/bin/<command_name>.py`
Custom command that:
- Reads SPL arguments
- Calls the API
- Returns results to the Splunk search pipeline

## Common Pitfalls to Avoid

1. **Don't hardcode credentials** - always use UCC's account management
2. **Don't forget checkpointing** - inputs should track last successful collection
3. **Don't ignore rate limits** - implement backoff/retry logic
4. **Don't skip error handling** - log errors properly for troubleshooting
5. **Don't violate AppInspect rules** - follow Splunk packaging standards

## Success Criteria

The TA is considered ready when:
- [ ] UCC build completes without errors
- [ ] Package installs cleanly in Splunk
- [ ] Configuration UI renders correctly
- [ ] Scripted input successfully collects data
- [ ] Custom commands (if any) return valid results
- [ ] AppInspect validation passes (or only minor/info findings)
- [ ] README includes clear setup instructions
- [ ] Basic tests pass

## Next Steps for Developer

After receiving the generated TA:
1. **Review** the generated code and configuration
2. **Test** in a Splunk development environment
3. **Refine** based on specific requirements or edge cases
4. **Deploy** to production or submit to Splunkbase
5. **Maintain** by updating API client code as the API evolves
